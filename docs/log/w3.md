### 3.2 OCR 调试全过程（commits `416b9be` → `0054435` → `2b26b6f`）

#### 3.2.1 PP-FormulaNet-S 公式识别准确度测试

**测试图片**：2022 年 9709 P1 Q1 试卷，crop 区域含 `f'(x)=2x^(-1/3)-x^(1/3)`。

**PP-FormulaNet-S 输出**（0.39s）：
```latex
\ {\mathrm{\tt~u c h~t h a t}}\,{\mathrm{f\tt~'}}(x)=2x^{-{\frac{1}{3}}}-x^{\frac{1}{3}}
.\,{\mathrm{\tt~I l~i s~g i v e n~t h a t}}\,{\mathrm f{\tt~f}}(8)=5.
```

**分析**：
- 核心公式 `2x^{-1/3}-x^{1/3}` ✅ 完全正确
- 但把 crop 中「such that」「it is given that」等英文字也转成了 `\mathrm{\tt~u c h~t h a t}` 乱码
- 6/6 质量指标通过（有分式、上标、2x、减号），但输出含不可读字符
- 原因：crop 区域太大，模型把文字+公式一起当成公式处理

**Qwen3-VL-8B 对比**（18.9s）：
```latex
$$\text{such that } f'(x) = 2x^{-\frac{1}{3}} - x^{\frac{1}{3}}. \text{ It is given that } f(8) = 5.$$
```
- 公式 **完美**，正确用 `\text{}` 包裹英文
- 慢 47 倍（18.9s vs 0.39s），但质量碾压
- **结论**：Qwen 做公式 OCR 比 PP-FormulaNet 强太多

#### 3.2.2 PaddleOCR 3.x API 兼容性修复

PaddleOCR 3.5.2 + paddlex 3.5.2 相比此前版本有大量 breaking change：

**Bug #1：`ocr.ocr()` 已废弃**
```python
# ❌ 旧代码，crash：
results = self.ocr.ocr(arr)  # DeprecationWarning, returns old format
text = line[1][0]            # IndexError on new format

# ✅ 新代码：
results = self.ocr.predict(arr)  # → list[OCRResult]
texts = r.get('rec_texts', [])    # dict 访问，非属性
scores = r.get('rec_scores', [])
polys = r.get('dt_polys', [])
```

**Bug #2：OCRResult 对象不是普通 dict**
- `OCRResult` 同时支持 `r['rec_texts']` 和 `r.rec_texts`
- 但 `hasattr(r, 'rec_texts')` 和 `'rec_texts' in r` 行为不一致
- 修复：统一用 `r.get('rec_texts', [])` 先试，然后 fallback 到 `getattr()`

**Bug #3：PPStructureV3 缺少依赖**
```python
# ❌ 直接 import 报错：
from paddleocr import PPStructureV3
engine = PPStructureV3()  # DependencyError: requires paddlex[ocr]
```
- 修复：`pip install "paddlex[ocr]"` → 安装了 beautifulsoup4, latex2mathml, openpyxl, tiktoken 等
- API 也变了：`engine(arr)` → `engine.predict(arr)`

**Bug #4：FormulaRecognition 缺少 ftfy 依赖**
```python
from paddleocr import FormulaRecognition
fr = FormulaRecognition(model_name='PP-FormulaNet-S')
fr.predict(arr)
# → ModuleNotFoundError: No module named 'ftfy'
```
- 修复：`pip install ftfy`
- ftfy 是 PP-FormulaNet 的 post-processor，用于修正 Unicode 编码问题

**Bug #5：FormulaRecognition 返回格式**
```python
# ❌ 全图传入公式识别，输出一行全是 \qquad：
result = fr(arr)  # TypeError: not callable
result = fr.predict(arr)  # OK, returns list[{rec_formula: ..., input_img: ...}]
```
- 需要 `ftfy` 库（PP-FormulaNet 的 post-processor 依赖）
- 修复：`pip install ftfy`

#### 3.2.3 预处理 Bug：图片变全黑

`ImagePreprocessor.preprocess(is_document=True)` 对数字试卷 PDF 做了阈值二值化：

```python
# 这段代码导致全黑（mean=0）：
gray = img.convert('L')
arr = np.array(gray)
threshold = np.percentile(arr, 50)
arr = ((arr > threshold) * 255).astype(np.uint8)  # 干净的 PDF 被全白→全黑
```

**根因**：阈值逻辑在像素均值 < 128 时会反色，然后以中位数为阈值。教材 PDF 底色是白色（255），文字是黑色。灰度均值 > 200，但代码设的阈值是 percentile(50)，相当于把所有像素和大多数白像素比较——只有极少数足够亮的像素才会被判定为 255，其余全部归 0。

**修复**：
- PaddleOCR：`preprocess(image_data, is_document=False)` → 只 resize，不做阈值
- Qwen VLM：传原始 `image_data`，完全不走预处理

#### 3.2.4 Qwen VLM Fallback → 直接主引擎

**v1 设计**（commit `0054435`）：PP-FormulaNet 先跑 → `_is_garbled_formula()` 检测 → 如果乱码 fallback 到 Qwen VLM。

**你的决策**：「现实使用无法评价普通 OCR 是否有正常工作，所以直接把有难度的工作都交给 Qwen，普通 OCR 只做最简单的 OCR 工作。」

**v2 设计**（commit `2b26b6f`）：彻底简化：

```
之前（70 行 code）：
  PP-FormulaNet → _is_garbled_formula() → Qwen VLM → MathPix
  PaddleOCR → PPStructureV3 → DECIMER → VisionAnalyzer → ChemistryOCREngine

之后（20 行 code）：
  PaddleOCR (0.5s) → 纯文字
  Qwen VLM (20-60s) → 全部其他内容
```

**移除的代码**：
- `_is_garbled_formula()` 乱码检测（4 种 pattern：`\mathrm{\tt}`, `\begin{dmath*}`, `\underline{\underline}`, 5×`qquad`）
- `_try_vlm_ocr()` fallback chain
- `ChemistryOCREngine` 从 `process_page` 中移除
- `PPStructureV3` 表格提取从 `process_page` 中移除
- `extract_graph_data()` 从 `process_page` 中移除
- `analyze_handwriting()` 从 `process_page` 中移除
- 全部条件路由逻辑（`if "circuit" in text` 等）

**新增**：`VisionAnalyzer.full_page_parse()` — 一个 prompt 覆盖全部内容类型：
```python
def _page_type_prompt(self, subject_code: str) -> str:
```

#### 3.2.5 PaddleOCR 文字提取准确度实测

| 原文 | PaddleOCR 输出 | Qwen VLM 输出 |
|------|---------------|---------------|
| `f'(x) = 2x^(-1/3) - x^(1/3)` | `f (x) = 2x-3 - x` (丢失上标) | `f'(x) = 2x^{-1/3} - x^{1/3}` (完美) |

| `[4]` | `[4]` (正确) | `[4]` (正确) |

PaddleOCR 的定位：纯英文字提取够用，分数/上标丢失但 Agent 能根据上下文理解。不做为主要内容提取工具。

#### 3.2.6 公式区域检测（commit `416b9be`）

新增 `PaddleOCREngine.detect_formula_regions()`：
```python
def detect_formula_regions(self, image_data: bytes) -> list:
    # 低置信度文本区域 + 数学符号匹配 → 可能是公式
    regions = self.extract_text(image_data)
    for r in regions:
        score = 0
        if r["confidence"] < 0.95: score += 2
        if math_pattern.search(r["text"]): score += 3
        if len(r["text"]) < 15 and has_digits: score += 1
        if score >= 3: formula_boxes.append(crop_bbox(r["bbox"]))
    return formula_boxes
```

这个功能在简化后暂时未使用（Qwen 全图解析替代了），但保留了以备 LM Studio 不可用时做 fallback。
