## §4 W4.5 Cost Logging + BudgetGuard

### Cost Logging

`_call_llm()` 从 API response 的 `usage` 字段提取：
- prompt_tokens
- completion_tokens
- reasoning_tokens（从 `completion_tokens_details.reasoning_tokens`）
- total_tokens
- latency_ms（从请求开始到结束）

费用计算：prompt_tokens / 1M * prompt_price + completion_tokens / 1M * completion_price。
USD → CNY 汇率 7.25。

### BudgetGuard

默认预算（可通过 env var 覆盖）：
- `BUDGET_MONTHLY_CNY=50`：月度 ¥50
- `BUDGET_DAILY_CNY=2`：每日 ¥2

超限策略：抛出 RuntimeError("预算限制: ...")，阻止后续 API 调用。
check_budget() 在每次 `_call_llm()` 前执行。

### CLI 命令

`/cost` 显示：
- 累计调用次数
- 累计 tokens
- 累计费用（¥ + $）
- 预算状态（正常/超限）
- 最近 7 天每日明细

`/stats` 显示：知识库统计 + 简要成本信息。

### 开发期费用

W4 开发测试实际消耗：~30 次 API 调用，~¥0.05。
成本极低因为：PedCoT 校准只有 15 个案例（每个 1-2 次调用），其余都是本地测试。

## §7 W4.1 评分校准 — 偏离原方案，重新设计

### 7.0A 创新架构：PedCoT 三路径评分引擎（核心创新）

这是 W4 最大的架构创新——不是简单修 prompt，而是从研究出发设计了一个
**三层降级评分引擎**，解决了「初见题目无 MS 仍能准确评分」的核心难题。

```
                        学生提交答案
                             │
                             ▼
                    ┌─────────────────┐
                    │  Path 0: MS 匹配  │  ← 搜索 ChromaDB past_papers (ms)
                    │  Reranker > 0.75? │     寻找真实 Cambridge mark scheme
                    └────────┬────────┘
                             │
                    ┌────────┴────────┐
                    │ YES             │ NO
                    ▼                 ▼
           ┌──────────────┐   ┌──────────────────────┐
           │ Path 1: 直接  │   │ Path 2: PedCoT 两阶段  │
           │ 评分         │   │                      │
           │              │   │ Stage 1 (Regenerate) │
           │ 1 次 API 调用 │   │ LLM 独立解题          │
           │ 对照真实 MS   │   │ → 概念/方法/计算/MAB  │
           │ M-A-B 逐点打分│   │                      │
           │              │   │ Stage 2 (Extract-     │
           │              │   │   Compare-Grade)     │
           │              │   │ 对照学生答案          │
           │              │   │ → M-A-B 逐点打分     │
           │              │   │ 2 次 API 调用        │
           └──────┬───────┘   └──────────┬───────────┘
                  │                      │
                  └──────────┬───────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │   输出 JSON      │
                    │  score_awarded   │
                    │  mark_points[]   │
                    │  rubric (4-dim)  │
                    │  misconception_  │
                    │    tags          │
                    └─────────────────┘
```

**为什么两阶段是必需的**（来自 PedCoT 论文 ablation）：
- 单阶段：LLM 同时看到题目+学生答案 → **conformality bias**（偷懒同意学生）
  → P+ 掉到 53%
- 双阶段：LLM 先独立解再对照 → 无偏判断
  → P+ 81%, R- 96%

**三路径对比**：

| 路径 | 触发条件 | API 次数 | 优点 | 缺点 |
|------|---------|---------|------|------|
| Path 1 直接评分 | MS 匹配成功（score > 0.75） | 1 | 对标真实考官标准，最快 | 依赖 MS 库覆盖 |
| Path 2 PedCoT | 无匹配 MS | 2 | 通用于任何题目，无偏判断 | 自动 MS 满分可能偏差 ±1 |
| Path 3 降级 | Stage 1/2 失败 | 0 | 保证系统不崩溃 | 返回 score=0 |

### 7.0B 原方案与失败原因

**原方案**：用已有的 `GRADING_JSON_PROMPT`（`agent/tutoring/prompts.py`），准备 15 道题 + gold scores，跑 grade() 对比。

**第一次尝试**：agent.grade() 返回 `JSONDecodeError: Expecting value: line 1 column 1 (char 0)`。
- 原因 1：`_call_llm()` 默认传入 `tools=TOOLS`，干扰 "Output valid JSON only" 指令，LLM 返回空 content
- 原因 2：`tools_list: list = None` 默认值时 `tools_list if tools_list is not None else TOOLS` = TOOLS，即「显式传 None = 用全部工具」，逻辑反了
- fix 1：改为 `tools_list: list = TOOLS`，显式传 `[]` 才无工具（commit dbc4220）

**第二次尝试**：agent.grade() 返回 `score_awarded: 10, score_max: 10`（题目满分只有 3 分）。
- 原因：prompt 中 `"score_max": <number>` 无上限约束，LLM 自己猜满分
- fix 2：在 prompt 中解析 mark scheme 的 `[N]` 标注推算满分，插入 `{max_marks}`（commit dbc4220）
- 结果：MAE=0.167, r=0.961 — PASS，但用的是合成 mark scheme，非真实材料

**第三次尝试（用户要求）**：必须用真实 Cambridge mark scheme 对照。
- 尝试 1：搜索 ChromaDB QP chunks → 内容是省略号（PyMuPDF text 提取失败，2022 试卷文本全是 `...`）
- 尝试 2：用 LM Studio VLM 逐页渲染 MS PDF → 成功提取 Q2 + 对应 MS，但每页 20-30s，19 页 MS 全扫需 10 分钟
- 尝试 3：搜索 ChromaDB MS chunks → 内容有编码损坏（``=`-`, ``=`=`, ``=`∫`）但可读
- 结论：当前 PDF 文本提取不行，需要真正的 QP+MS 对照数据集，或者用 VLM 批量提取

**此时用户提出更深层的问题**：当一道初见的题匹配不到完全对应 MS 时，如何确保仍得到严谨批改？

### 7.1 深度研究（4 轮 web search）

进行了系统性的文献研究，覆盖：

**Cambridge 官方评分体系**（来源：官网 MS PDF，papacambridge, studocu, excelas.ai）
- M-A-B 三分法：Method（方法）/ Accuracy（精确）/ Bonus（独立）
- M→A 依赖：A 分必须在 M 分获得后才能给
- FT（Follow-Through）：前序结果错但后续方法对，照给 M 分
- 正向评分：只加分不扣分，只给整数分，错写不罚
- ISW（Ignore Subsequent Working）：正确后的错误忽略
- 3 s.f. 标准，角度到 1 d.p.
- 关键发现：M 分不因算术错误扣分，但必须「把方法应用于具体问题」，只引用公式不够

**自动化评分研究**：

| 论文 | 方法 | 结果 | 相关性 |
|------|------|------|--------|
| MathMistake Checker (arXiv 2503.04291) | OCR + LLM PedCoT 两阶段 | 无参考答案的开放式评分 | 直接可借鉴 |
| PedCoT (Jiang et al. IJCAI 2024) | 两阶段：Regenerate → Extract-Compare | MF Acc 83% vs baseline 74% | **核心参考** |
| ProofGrader (arXiv 2510.13888) | LLM + 参考方案 + MS 作为 context | MAE 0.926/7 | MS 质量很关键 |
| GPT-5 Calculus (arXiv 2510.05162) | 置信度过滤 + IRT 风险 | R²=0.95（严格过滤后） | 置信度过滤思路 |
| Pensieve Grader (arXiv 2507.01431) | 端到端：转录→评分→置信度→反馈 | 65% 时间节省 | 全流程参考 |
| Rubric Injection (D19-6119) | 将 rubric key elements 注入 SAG | 低资源下显著提升 | rubric 生成思路 |

**关键发现：PedCoT 两阶段必要性**

Jiang et al. 做了 ablation：单阶段 vs 双阶段。
- 单阶段：LLM 同时看到问题和学生答案 → conformality bias（LLM 偷懒，倾向于同意学生已有答案），P+ 从 81.25% 掉到 53.33%
- 双阶段：Stage 1 LLM 不看学生答案独立解题 → Stage 2 对照评分 → P+ 81.25%, R- 96.22%
- **结论：必须两阶段**。先让 LLM 自己解一遍，再对照。这个发现直接指导了我们的评分重构。

### 7.2 实现：PedCoT 三路径评分引擎

创建 `agent/grading/grader.py` — 替换原来的 `agent/tutoring/prompts.py:GRADING_JSON_PROMPT`。

**路径 1：真实 MS 匹配**
```
question → search_past_papers(ms, rerank=True) → if score > 0.75
  → 直接用真实 MS 评分（1 次 API 调用）
```
搜索 ChromaDB past_papers collection (paper_type="ms")，Reranker 阈值 0.75。

**路径 2：PedCoT 两阶段**
```
if 无匹配 MS:
  Stage 1 (Regenerate): LLM 只看到 question → 生成:
    [概念] [方法步骤] [计算过程] [M-A-B 分配表]
  
  Stage 2 (Extract-Compare-Grade): LLM 看到 question + Stage-1 输出 + student_answer → 
    → 提取学生实际步骤
    → 逐步与 Stage-1 预期对照
    → 按 M-A-B 规则打分
    → 输出 JSON {score_awarded, mark_points[], rubric, ...}
```

**路径 3：降级**（Stage 1 或 Stage 2 失败 → 返回 fallback `score=0, tags=["system_error"]`）

**PedCoT Stage 1 Prompt 设计**（基于 Bloom 认知模型的三个层次）：
1. **Remember（记忆）**：列出涉及的数学概念
2. **Understand（理解）**：分解解决方法步骤
3. **Apply（应用）**：执行计算并分配 M-A-B 分

**PedCoT Stage 2 Prompt 设计**：
- 包含完整的 Cambridge 评分规则（M/A/B/FT/DM/ISW/3sf/正向评分）
- 逐 mark point 判断：student earned this mark? why/why not?
- 应用 M→A 依赖：M 未得则 A 不得
- 应用 FT：前序错但方法对 → M 分照给
- 输出 JSON 含 mark_points[] (逐分说明)

- **MAE = 0.300 分** (< 0.5 ✓ PASS)
- **Pearson r = 0.898** (> 0.85 ✓ PASS)

| DIF-03 | 1.5/3 | 0/3 | +1.5 | 代理偏严。学生写了 5(2x+1)^4 但少了乘 2，给了 0。应为 1-2 分 |
| MEC-01 | 2/2 | 2/2 | 0.0 | 完美匹配 |
| MEC-02 | 0/2 | 0/2 | 0.0 | 全错→0 分，正确 |
| MEC-03 | 1.5/2 | 2/2 | -0.5 | 代理偏宽松。极简答案 "a=6t-4=8" 给了满分，应为部分分 |

**偏差模式总结**：

1. **自动 MS 满分偏差**（P1-04）：Stage-1 自动生成的满分比 Cambridge 官方少 1 分。自动 MS 合并了某些 mark points（如 A1+A1 合并为单个 A1）。
2. **代理偏严**（DIF-03）：当学生答案缺少中间步骤时，代理倾向于 0 分而非部分分。Cambridge 考官会给 partial credit 如果能看到部分正确方法。
3. **代理偏松**（INT-03, MEC-03）：当学生有尝试意图但结果不对时，代理倾向给同情分。Cambridge 考官更严格。
4. **10/15 完全匹配**：在答案明确对错（有完整过程且结果对、有完整过程且结果错、无过程且答案错）的案例中代理表现优秀。

### 7.4 限制和后续

**当前限制**：
- 自动 MS 满分可能与 Cambridge 官方不一致（±1 分），需要更大的 MS 匹配库
- MS 匹配率不够高：ChromaDB 中 MS chunks 有编码损坏（``, ``），影响匹配质量
- QP 文本提取失败（2022 试卷全是省略号），无法做 QP→MS 对照
- 真人评分对照缺失：gold set 是人工基于 Cambridge 规则设定的，不是真人评分
- 15 个案例不够有代表性：需要扩展到 50+ 案例，涵盖更多 topic、更多题型

**建议后续（W5+）**：
1. 用 VLM 批量提取 QP+MS 对照集（离线批处理，每页 20-30s，可忍受）
2. 建立 question→MS 直接映射表（200+ 题），超过 Reranker 匹配
3. 收集或模拟真人学生答案（错误类型多样化），找有 A-Level 经验的真人评分做对照
4. 对 PedCoT Stage-1 的输出做质量检查（MS 满分是否合理）
5. 引入置信度阈值 deferral（如 ProofGrader/GPT-5 论文中，低置信度路由到人工）
