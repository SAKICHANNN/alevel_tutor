## §13 用户反馈修复（2026-06-01）

### Bug 22: Gradio 4.44.1 + Python 3.9 schema 类型 bug

**症状**：启动 `web/app.py` 时报 `TypeError: argument of type 'bool' is not iterable`，来自 `gradio_client/utils.py:863` 的 `get_type()`。Gradio 4.44.1 在 Python 3.9 上遇到 bool schema 时崩溃。Gradio 5.x 需要 Python 3.10+。

**修复**（47b89f9）：在 `web/app.py` 顶部 monkey-patch `gradio_client.utils.get_type()`：
- 若 schema 是 bool 类型，返回 `"boolean"`
- 否则委托原始函数
- 必须在 `import gradio` 之前执行

### Bug 23: DeepSeek 自我纠错过程泄漏到学生输出

**用户反馈**：回答中出现「流入 = 5A，流出 = 3A + 4A = 7A ❌ 不对！实际上...等等，重新算」——LLM 的内部草稿纠错过程直接展示给学生，且 ASCII 电路图错误。

**根因分析**：
1. DeepSeek V4（94% 幻觉率）在长篇回答中「边写边纠正」
2. `tutor` 模型 `thinking=False`——没有独立的推理链，LLM 的思考过程直接写入 `content`
3. ASCII 艺术图是 LLM 的固有弱点——公式对但图编的

**修复**（2631b19 + 9599798），三层防护：

| 层 | 文件 | 措施 |
|----|------|------|
| System Prompt | prompts.py | 新增 `⛔ 绝对禁止` 章节：禁止输出纠错标记（不对/等等重新算）、禁止编造不确认的公式、禁止展示工具调用过程 |
| 后处理过滤 | core.py | `_sanitize_output()` — 正则检测自我修正模式 → 在回答前加 ⚠️ 警告 |
| Thinking 模式 | config.py | `tutor.thinking = True` — 启用独立推理链，推理在 `reasoning_content` 中、答案在 `content` 中，互不干扰 |
| Reasoning 连续性 | core.py | `chat()` 保存 `reasoning_content` 到 conversation history，确保多轮对话 + tool call 不触发 HTTP 400 |

### Bug 24: Mermaid.js 图表渲染失败（5 次尝试）

> commits 6ec7abf, b8836a9, 53564c5, de9d266, e357a65, 6af0756, 9b385f5

**用户反馈**：LLM 输出的 ASCII 电路图排版错乱，「mermaid 还是不行，不能正常显示而是显示了代码」。

**尝试 1：系统 Prompt 引导 → 失败**（6ec7abf）
在 prompt 中添加 Mermaid 示例，期望 LLM 自觉输出 `` ```mermaid `` 代码块。
结果：LLM 能生成 Mermaid 代码，但 Gradio 将其显示为源代码（Gradio 4.44.1 不含 Mermaid.js 渲染器，PR #10854 未进入此版本）。

**尝试 2：CDN 注入 + MutationObserver → 失败**（53564c5 + de9d266 + e357a65）
通过 `gr.Blocks(head=...)` 注入 Mermaid.js CDN，用 MutationObserver 扫描 DOM 转换 `<code class="language-mermaid">` → `<pre class="mermaid">`，调用 `mermaid.run()` 渲染。
失败原因：
- `head` 参数在 Gradio 4.44.1 中不可靠
- `gr.HTML(visible=False)` 注入的脚本可能执行时机不对
- MutationObserver 找不到 Gradio Chatbot 生成的 DOM 元素
- 即使脚本成功注入，浏览器 F12 Console 无报错但不渲染

**尝试 3：Kroki API 服务端渲染 → 逻辑正确但缓存导致失败**（6af0756）
改用 `_render_mermaid_blocks()` — 服务端 POST Mermaid 代码到 `https://kroki.io/mermaid/svg` → 返回 SVG → base64 编码 → 嵌入 `<img>` 标签。
问题：Python `__pycache__` 缓存了旧版 `_sanitize_output`（不含 `_render_mermaid_blocks` 调用）。清除 `__pycache__` 后函数调用恢复正常。

**尝试 4：原始 HTML `<img>` → Safari 兼容失败**（6af0756）
Kroki API 渲染成功，但 Safari 上 `<img src="data:image/svg+xml;base64,...">` 不显示。
原因：Gradio Chatbot 的 markdown 渲染器（marked.js）对原始 HTML 标签进行安全过滤，`<img>` 被 strip 掉。

**尝试 5：Markdown 图片语法 → 成功**（9b385f5）
改用标准 Markdown 图片语法：`![diagram](data:image/svg+xml;base64,...)`
结果：**所有浏览器正常显示**。Markdown 解析器原生支持此语法，不做过滤。

**最终方案**（7 个 commit）：
1. System prompt 包含 Mermaid 示例 + ASCII 禁止规则
2. `_render_mermaid_blocks()` 服务端渲染（Kroki API，~4s）
3. `_sanitize_output()` 调用 `render_all_diagrams()`（`agent/diagrams/renderer.py`）
4. 输出使用 Markdown `![diagram](base64SVG)` 语法

### Bug 25: LaTeX 公式不渲染

**用户反馈**：「表格里公式显示错误」。

**根因**：
1. Chatbot `latex_delimiters` 只配了 `$$`（独立公式），缺少 `$`（行内公式），表格中的 `$E=mc^2$` 不渲染
2. System prompt 的 `base = f"""...` 是 f-string，LaTeX 的 `{...}` 花括号被 Python 当成 f-string 表达式解析，导致 `SyntaxError`

**修复**（6af0756）：
- Chatbot 添加 `{"left": "$", "right": "$", "display": False}`
- `base = f"""` → `base = """`，subjects_summary 单独 f-string
- System prompt 新增 `LaTeX 数学公式规范` 章节（常用命令、表格写法、禁止 Unicode 伪公式）

### Bug 26: Python `__pycache__` 缓存旧字节码

**症状**：`_render_mermaid_blocks` 函数代码正确但完全不执行，`_sanitize_output` 输出仍含原始 Mermaid 代码块。

**根因**：修改 `_sanitize_output` 后添加 `_render_mermaid_blocks()` 调用，但项目目录下的 `__pycache__` 保存了旧版 `.pyc` 字节码。Python 加载缓存而非重新编译源文件。

**修复**：`find . -type d -name __pycache__ -exec rm -rf {} +`

**教训**：每次修改函数的调用关系后应清理缓存，或设置 `PYTHONDONTWRITEBYTECODE=1`。

## §16 图表精度革命——预置模板系统（2026-06-01）

### 16.1 问题诊断

**用户反馈**：「经济学复合曲线图有严重位置错误，大体在模仿正确的样子但相差甚远。」

**深度研究**：
- DeTikZify (arXiv 2405.15306)：即使 360k 训练样本，LLM 生成的 TikZ 仍有 fidelity errors（交点不准、比例失调）
- CAGE (arXiv 2604.09691)：代码生成精度高（96% LEM）但视觉质量差 → 需要「code anchors + diffusion refinement」
- TeX StackExchange：用户求助「TikZ 图不匹配原图」，回答用 Metapost 重写——根本原因是 LLM 不懂数学坐标

**根本结论**：LLM 不具备生成精确数学坐标的能力。这是一个已知的学术难题。

### 16.2 解决方案

**预置模板库**——LLM 只负责选模板和填标签，不生成坐标：
```
LLM 输出: ```tikz template=econ:ad-as```
      ↓
服务端: 加载预验证 TikZ → 填默认标签 → Kroki 渲染
      ↓
结果: 精确交点、正确斜率、标准标注
```

### 16.3 模板库

| 科目 | 模板数 | 覆盖图 |
|------|--------|--------|
| 9708 Economics | 9 | 供需均衡、需求右移、负外部性(DWL)、AD-AS、AD 扩张、税收归宿、PPC、最高限价、凯恩斯 LRAS |
| 9702 Physics | 5 | 串联电路、并联电路、自由体图、波形图、斜面力分解 |

**精确性保证**：每个模板的交点都通过联立方程计算：
- `demand-supply`: D=-x+6, S=0.8x+0.5 → E=(3.06, 2.94)
- `ad-as`: AD=-0.9x+7.5, SRAS=0.5x+2 → E=(3.93, 3.96)

## §18 KaTeX Parse Error 排查与修复（2026-06-01）

> Commits: 3d475eb, c8ec30a

### 18.1 症状

Gradio Chatbot 显示 "KaTeX parse error" 而非公式渲染。

### 18.2 根因

| # | 问题 | 示例 | KaTeX 行为 |
|---|------|------|-----------|
| 1 | LLM 用 `$` 当货币符号 | `价格 $10` | 把 `$10` 当数学块起始，无配对 `$` → 报错 |
| 2 | `&` 在 `$...$` 内部 | `$A & B$` | KaTeX 将 `&` 视为对齐符，不允许在行内公式中 |
| 3 | `\ce{}` 命令 | `$\ce{H2O}$` | KaTeX 不支持 mhchem 扩展 |

### 18.3 修复：双层防线

**① 系统提示层** (`prompts.py`)：
- 明确禁止 4 项：`$` 当货币（写"10元"）、`&` 在 `$...$` 中、`\ce{}`、不配对 `$`
- 正反面示例

**② 后处理层** (`core.py:_fix_katex`)：
- 逐字符扫描器（非正则），正确处理 `$$a = $b$ + c$$` 嵌套
- 预处理：`$10` → `10元`（匹配 `$`+数字，保留 `$10$`）
- LaTeX 块内：`&` → `\text{ and }`、`\ce{...}` → `\text{...}`
- LaTeX 块外：游离 `$` → `＄`（全角，不作分隔符）

**③ 测试**：9 项手动测试通过（货币、行内公式、独立公式、混合、游离、嵌套、`&`修复、`$10$`保留、千分位），未计入自动化测试套件

### 33.4 重复检索智能截停（877ac64 + 8d5685e + 4c0fbbe）

**问题**：去掉硬限制轮数后，LLM 在连续几轮中重复调用 `search_textbook("Le Chatelier")` 完全相同的查询。

**方案演进**：

| commit | 方案 | 问题 |
|--------|------|------|
| 877ac64 | 全历史去重——所有已调用过的 tool+args 再出现就截停 | 误杀合法重复（不同轮次搜同一个概念可能是合理的） |
| 8d5685e | 改回连续重复——只检查与上一轮最后一个 call 是否相同 | A+B → 首项 A 匹配即全停，B 丢失 |
| 4c0fbbe | **最终方案**：逐项过滤——遍历本轮 tool_calls，重复项跳过，新项保留。全部都是重复时才截停 | 正确 |

**最终逻辑**：
```python
filtered = []
for tc in msg["tool_calls"]:
    tc_key = (tc["function"]["name"], tc["function"]["arguments"])
    if tc_key == _last_tc_key:
        continue  # 跳过重复，保留其他有效调用
    _last_tc_key = tc_key
    filtered.append(tc)
if not filtered:  # 全部重复 → 提示用户细化问题
    return "⚠️ 检测到全部重复检索..."
msg["tool_calls"] = filtered
```

**关键行为**：
| 轮次 | LLM 调用 | 结果 |
|------|---------|------|
| 1 | search_textbook("X") | ✅ 执行 |
| 2 | search_textbook("X") + get_exam_pattern("Y") | ❌ 跳过 X，✅ 执行 Y |
| 2 | search_textbook("X")（只有重复） | 🛑 截停，提示用户 |

## §34 Bug 集中修复（2026-06-02，基于 debug log 分析）

> commit 580983c

### Bug A: 全局追踪变量始终为默认值
**根因**：`web/app.py` 用 `from agent.tutoring.core import _last_model` 导入——创建的是本地引用。当 `chat()` 里 `global _last_model; _last_model = "deepseek"` 重新绑定了模块级名字，`app.py` 的本地引用仍指向旧值 `""`。

**修复**：改为 `import agent.tutoring.core as _core`，所有追踪变量通过 `_core._last_model` 访问。这样始终取当前模块级值。

### Bug B: LLM 不调用工具直接回答
**根因**：Prompt 用「必须执行」措辞不够强制。LLM 在 thinking 模式下有时跳过 tool_calls。

**修复**：提示词改为「每次回答前必须先调用」「在 tool_calls 返回之前不要生成任何 content」「不调用工具属于违规」的强硬表述。

### Bug C: TikZ 代码块静默渲染失败
**根因**：`render_mermaid/tikz` 的 `except Exception: pass` 吃掉所有错误。`render_all_diagrams` 无错误统计。

**修复**：`render_all_diagrams` 改为 try/except 包裹每个子渲染器，异常时计数并通过 stderr 输出 `[diagrams] N renderer(s) failed`。

### Bug D: 自我修正警告误触发
**根因**：正则 `(?:❌|不对|等等|重新算|实际上|查到了)` 在 TikZ 代码块中误匹配。电路图 LaTeX 代码包含 `\text{...}` 和特殊字符。

**修复**：（1）正则收紧为 `不对，|等等，重新|重新算一下|查到了，`，要求更具体的上下文。（2）检测前先 `_CODE_BLOCK_RE.sub('', content)` 排除代码块。

### Bug E: Debug log 截断回复
**修复**：`response[:5000]` → `response[:15000]`。

### Bug F: 货币 $ 符号
已在 post-processing 中通过 `_fix_katex` 处理，本次不改。
