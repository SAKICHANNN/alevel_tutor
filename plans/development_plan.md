### ⚠️ 已写代码但未验证（有 bug 风险）

| 模块 | 风险 |
|------|------|
| OCR 管道 (`ocr_pipeline.py`) | 6 个引擎都未对真实试卷图片测试过。Surya 调用 API 可能错误、DECIMER import 路径未确认、PaddleOCR 首次加载需下载 500MB 模型 |
| 知识库构建 (`kb_builder.py`) | chunking 策略对数学/化学公式不适配（用 `[.!?]` 分句会切断公式）、embedding fallback 到 all-MiniLM 精度不够 |
| 向量检索 (`retriever.py`) | fallback 模式用的缓存无界（内存泄漏）、techniques collection 构建了但从未被检索 |
| Agent 工具调用 (`core.py`) | `grade_homework_image` 工具永远返回静态文本，没有真正调用批改；没有 token 计数 |
| Vision (`vision.py`) | 只有 GLM-4V/Qwen-VL 两种 provider，不含 GPT-4V；图片无压缩 OOM 风险 |
| Pattern 数据库 (`patterns.py`) | 只有 10 个套路模板，覆盖 syllabus 不到 5%；`get_pattern` 匹配逻辑简单 |

### ❌ 严重缺失

| 缺失项 | 影响 |
|--------|------|
| **OCR 管道端到端未验证** | 整个「图片识题→结构化→批改」流程走不通 |
| **ChromaDB 未构建** | RAG 搜索功能不可用，只能靠 LLM 内建知识回答 |
| **批改精度无基准测试** | 不知道 AI 批改到底准不准，无法上线 |
| **无 Web 前端** | 只有 CLI，学生无法使用 |
| **无数据库** | 对话不持久化、错题不记录、用户无画像 |
| **无 API 降级/费用监控** | 预算 ¥20/月 会轻易超支 |
| **手写识别未实测** | 学生手写作业批改是最核心需求，但一条链都还没跑通 |

| **1.5** 实现混合搜索（语义 + 关键词） | 2h | 1.4 | `retriever.py` 增加 BM25/关键词搜索 + 语义搜索融合 |
| **1.6** 修复 `search_techniques` 方法 | 1h | 1.3 | 增加对 techniques collection 的搜索（目前构建了但没查询接口） |

## 三、优先级依赖图

```
Phase 0 (验证回路) ──→ Phase 1 (知识库) ──→ Phase 2 (套路扩充)
         │                    │                    │
         └──→ Phase 3 (OCR+批改) ─────────────────┘
                       │
                       └──→ Phase 4 (Web前端) ──→ Phase 5 (数据库) ──→ Phase 6 (成本控制) ──→ Phase 7 (Pilot)
```

- Phase 0-3 可以在 4 周内并行做好（不同的人/时间块）
- Phase 4 需要 Phase 3 完成（后端接口稳定）
- Phase 5-7 串行依赖

## 四、推荐执行方式

如果你一个人开发：

```
Week 1-2: Phase 0 (验证) + Phase 1 (知识库)
Week 3:   Phase 2 (套路)  + Phase 3 (OCR优化)
Week 4-5: Phase 4 (Web前端)
Week 6:   Phase 5 (数据库) + Phase 6 (成本控制)
Week 7-8: Phase 7 (Pilot)
```

总计 **8 周** 可以从当前代码状态推进到校内 Pilot。

如果有另一个人帮忙前端，Phase 4 可以和 Phase 2/3 并行：
```
Week 1-2: Phase 0+1 (后端)
Week 3-4: Phase 2+3 (套路+OCR) || Phase 4 (前端)
Week 5:   Phase 5+6 (数据库+成本)
Week 6-8: Phase 7 (Pilot)
6 周到 Pilot。
```
